**Researched:** 2026-02-05
**Domain:** FastAPI startup automation with entity seeding
**Confidence:** HIGH

## Summary

Phase 3.1 requires integrating entity seeding into FastAPI's lifespan context manager to ensure entities exist before the scheduler starts processing articles. This is a critical integration fix that resolves the blocking gap where foreign key constraints prevent article ingestion from working end-to-end.

**Primary recommendation:** Use FastAPI lifespan context manager with proper dependency injection to seed entities before scheduler startup, ensuring database integrity and maintaining startup sequence dependencies.

### Pattern 1: Lifespan Context Manager with Entity Seeding
**What:** Use FastAPI's lifespan context manager to ensure entities are seeded before scheduler starts
**When to use:** Always for this requirement - follows modern FastAPI patterns and ensures proper startup sequence
**Example:**
```python
from contextlib import asynccontextmanager
from fastapi import FastAPI
from db.session import AsyncSessionLocal
from scripts.seed_entities import seed_entities

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup - seed entities before scheduler
    print("Seeding entities...")
    await seed_entities()
    print("Entity seeding complete")

    yield  # Application runs here

    # Shutdown - cleanup handled by existing pattern
    print("Shutting down...")

app = FastAPI(lifespan=lifespan)
```

### Anti-Patterns to Avoid
- **Startup order violations:** Don't start scheduler before entities are seeded
- **Sync database operations:** Use async database operations in lifespan context
- **Manual script execution:** Never require manual intervention during startup

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Database seeding logic | Custom insert statements | Existing seed_entities script | Already implements deduplication and error handling |
| Startup sequence management | Manual try/catch chains | FastAPI lifespan context manager | Built-in resource cleanup and error handling |
| Entity validation | Custom checking functions | Entity models and constraints | Database-level validation is more reliable |

**Key insight:** The existing `seed_entities.py` script is already well-designed and should be reused rather than reimplementing seeding logic.

## Common Pitfalls

### Pitfall 1: Database Connection Timing
**What goes wrong:** Entity seeding attempts before database is fully initialized
**Why it happens:** Missing migration step or database connection pool not ready
**How to avoid:** Run migrations first (existing pattern), then seed entities
**Warning signs:** "connection refused" errors during seeding

### Pitfall 2: Duplicate Entity Creation
**What goes wrong:** Seeding script creates duplicates on each startup
**Why it happens:** Script doesn't check existing entities properly (but it already does)
**Warning signs:** Unique constraint violation errors

### Pitfall 3: Scheduler Startup Before Entities
**What goes wrong:** Jobs start processing articles before entities exist
**Why it happens:** Wrong startup sequence or missing dependencies
**Warning signs:** Foreign key constraint violations in job logs

### Pitfall 4: Silent Failures in Seeding
**What goes wrong:** Entity seeding fails but application continues
**Why it happens:** No proper error handling or logging in integration
**How to avoid:** Add try/catch with appropriate error messages and logging
**Warning signs:** Empty entities endpoint after startup

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI
from db.session import AsyncSessionLocal
from scripts.seed_entities import seed_entities
from pipeline.scheduler import setup_jobs, scheduler
import structlog

logger = structlog.get_logger(__name__)

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup sequence
    logger.info("Starting up application...")

    # Run migrations first (existing)
    run_migrations()

    # Seed entities before scheduler
    try:
        logger.info("Starting entity seeding...")
        await seed_entities()
        logger.info("Entity seeding completed successfully")
    except Exception as e:
        logger.error("Entity seeding failed", error=str(e))
        raise  # Fail startup if seeding fails

    # Setup and start scheduler
    setup_jobs()
    logger.info(f"Registered {len(scheduler.get_jobs())} scheduled jobs")
    scheduler.start()
    logger.info("APScheduler started")

    yield  # Application runs here

    # Shutdown sequence (existing)
    logger.info("Shutting down application...")
    scheduler.shutdown(wait=True)
    logger.info("APScheduler shutdown complete")
```

## Open Questions

1. **Seeding error handling strategy**
   - What we know: Existing script handles duplicates gracefully
   - What's unclear: Should seeding be fatal to startup or should we continue?
   - Recommendation: Make seeding fatal to startup - empty entity database breaks the entire system

2. **Multi-process considerations**
   - What we know: FastAPI with uvicorn can run multiple workers
   - What's unclear: Will seeding run in all workers?
   - Recommendation: This might require locking in production

3. **Seeding idempotency**
   - What we know: Script already handles idempotency
   - What's unclear: Should we add seeding status tracking?
   - Recommendation: Keep current simple approach
